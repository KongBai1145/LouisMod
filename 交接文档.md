## 项目概况 / Project Overview

LouisMod 是一个外部、只读的 CS2 gameplay enhancer（基于 Valthrun 二次开发）。本 Fork 在原版基础上扩展了多项功能，并重新设计了 Web 控制面板。

**LouisMod is an external, read-only CS2 gameplay enhancer forked from Valthrun, with enhanced features and a redesigned web control panel.**

**技术栈 / Tech Stack:**
- 语言 / Language：Rust (nightly toolchain)
- 驱动接口 / Driver：`louismod-kdriver` — 自研内核驱动 + 用户态间接 syscall 双后端 / custom kernel driver + user-mode indirect syscall dual backend
- 双后端自动降级 / Auto-fallback：内核驱动可用时优先使用，不可用时自动回退用户态 / kernel preferred, user-mode on failure
- GUI Overlay：imgui-rs + 自定义 OpenGL 渲染
- Web 面板 / Panel：Axum (HTTP/WS) + 纯前端 SPA / vanilla frontend SPA
- Schema 解析 / Resolution：自定义 `raw_struct` + 运行时偏移量解析 / runtime offset resolution

## 核心架构 / Core Architecture

### 0. 双后端系统 / Dual Backend System

LouisMod 使用**双后端自动降级**架构。`create_driver()` 工厂函数按顺序尝试：

**LouisMod uses a dual-backend auto-fallback architecture. The `create_driver()` factory tries in order:**

```
create_driver()
  ├── 1. LouisModDriver::create_from_env()  ── 内核驱动 (需要 louismod.sys 已加载)
  └── 2. UserModeDriver::create()           ── 用户态间接 syscall (无驱动时自动回退)
```

**内核驱动后端 / Kernel Driver Backend (LouisModDriver):**
- 通过 `CreateFileW("\\.\LouisMod")` + `DeviceIoControl` 与内核驱动通信
- 提供：进程列表、模块列表、内存读写、键盘/鼠标输入、进程保护(ObRegisterCallbacks)、CR3 缓解
- 需要：testsigning 开启 + 驱动已加载 + 管理员权限
- **Communicates with louismod.sys via DeviceIoControl. Full kernel-level stealth with ObRegisterCallbacks and CR3 mitigation.**

**用户态后端 / User-Mode Backend (UserModeDriver):**
- 使用间接 syscall（从磁盘 ntdll.dll 提取 syscall 号）
- syscall 返回地址伪装：`call` 到 ntdll 的 `syscall;ret` gadget
- 进程枚举：间接 NtQuerySystemInformation(SystemProcessInformation)
- 模块枚举：PEB walk (间接 NtQueryInformationProcess + NtReadVirtualMemory)
- 句柄：间接 NtOpenProcess（绕过 kernel32/ntdll 函数入口 hook）
- 输入：SendInput（用户态唯一可行方案）
- 仅需：非管理员即可运行
- **Indirect syscalls with return-address camouflage. No admin, no driver, no reboot required. Sufficient for VAC.**

### DriverInterface trait

两套后端通过统一的 `DriverInterface` trait 向上层暴露相同接口 / **Both backends expose identical interface via DriverInterface trait:**

```rust
pub trait DriverInterface: Send + Sync {
    fn driver_features(&self) -> DriverFeature;
    fn driver_version(&self) -> VersionInfo;
    fn total_read_calls(&self) -> usize;
    fn list_processes(&self) -> IResult<Vec<ProcessInfo>>;
    fn list_modules(&self, pid: ProcessId, dt: DirectoryTableType) -> IResult<Vec<ProcessModuleInfo>>;
    fn read_bytes(&self, pid: ProcessId, dt: DirectoryTableType, addr: u64, buf: &mut [u8]) -> IResult<()>;
    fn write_bytes(&self, pid: ProcessId, dt: DirectoryTableType, addr: u64, buf: &[u8]) -> IResult<()>;
    fn send_keyboard_state(&self, states: &[KeyboardState]) -> IResult<()>;
    fn send_mouse_state(&self, states: &[MouseState]) -> IResult<()>;
    fn toggle_process_protection(&self, mode: ProcessProtectionMode) -> IResult<()>;
    fn add_metrics_record(&self, rt: &str, rp: &str) -> IResult<()>;
}
```

上层通过 `Arc<dyn DriverInterface>` 使用，无需关心底层是驱动还是用户态。

### DriverFeature 标志位 / Feature Flags

```rust
pub struct DriverFeature(u32);
// PROCESS_LIST | PROCESS_MODULES | MEMORY_READ | MEMORY_WRITE
// | INPUT_KEYBOARD | INPUT_MOUSE | METRICS
// 内核特有 / Kernel-only: PROCESS_PROTECTION_KERNEL | CR3_SHENANIGANS
```

### 1. Enhancement 系统 / Enhancement System

所有游戏功能都实现 `Enhancement` trait：

```rust
pub trait Enhancement {
    fn update(&mut self, ctx: &UpdateContext) -> anyhow::Result<()>;
    fn render(&self, states: &StateRegistry, ui: &imgui::Ui, ...) -> anyhow::Result<()>;
    fn render_debug_window(&mut self, ...) -> anyhow::Result<()> { Ok(()) }
}
```

- **update()** — 每帧调用，从游戏内存读取数据并执行逻辑
- **render()** — ImGui 渲染，用于 Overlay 显示

所有 Enhancement 在 `main.rs` 中注册并循环调用。

### 2. 状态注册表 / StateRegistry

```rust
let mut app_state = StateRegistry::new(1024 * 8);
app_state.register::<StateCS2Memory>()?;
app_state.register::<StateEntityList>()?;
```

各模块通过 `ctx.states.resolve::<T>(())?` 获取共享状态。

### 3. CS2Handle 与内存读写 / Memory I/O

- `read_sized<T>(address)` — 读取任意类型 (≤64 bytes: 栈分配 / stack; >64: 堆 / heap)
- `read_slice<T>(address, buffer)` — 批量读取 (≤256 bytes: 栈 / stack; >256: 堆 / heap)
- `write_sized<T>(address, value)` — 写入游戏内存 (仅 silent_aim 使用)
- `send_mouse_state()` / `send_keyboard_state()` — 发送输入事件
- `create_memory_view()` — 创建 MemoryView 供 schema trait 读取

### 4. Schema 与偏移量解析 / Offset Resolution

**两种偏移量获取方式 / Two offset resolution methods：**

1. **Trait 方法** (推荐读取 / recommended for reads)：
   ```rust
   let angles = pawn.m_angEyeAngles()?;
   ```

2. **`runtime_offset!` 宏** (写入时使用 / for writes)：
   ```rust
   let offset = runtime_offset!(15824, "client.dll", "C_CSPlayerPawn", "m_angEyeAngles");
   ```
   第一个参数是 fallback 默认值。宏内部创建 `static mut Option<u64>` 缓存。

## 用户态后端详解 / User-Mode Backend Details

### Syscall 号提取 / Syscall Number Extraction

`syscall::load_syscall_table()` 从磁盘 `C:\Windows\System32\ntdll.dll` 解析 PE 结构 + export 目录，提取 5 个关键 syscall 号。提取方法：定位 export 函数 RVA → 检查 stub 第一字节是否为 `0xB8` (mov eax, imm32) → 读取 imm32。

**Parses ntdll.dll from disk (not the in-memory copy that anti-cheats may hook). Reads PE export directory, locates each Nt* function stub, checks for `0xB8` opcode, extracts the syscall number.**

| 函数 / Function | 用途 / Purpose |
|------|------|
| NtReadVirtualMemory | 跨进程读取 / Cross-process read |
| NtWriteVirtualMemory | 跨进程写入 / Cross-process write |
| NtQuerySystemInformation | 进程枚举 / Process enumeration |
| NtOpenProcess | 打开进程句柄 / Open process handle |
| NtQueryInformationProcess | 获取 PEB 地址 / Get PEB address |

### Syscall 返回地址伪装 / Return Address Camouflage

**问题 / Problem:** 直接在我们模块执行 `syscall`，内核回溯显示调用来自 `controller.exe`。

**解决方案 / Solution:** 在已加载 ntdll.dll 的 `.text` 段中搜索 `0F 05 C3` (syscall; ret) 字节序列。通过 `call` 跳转到该地址执行 syscall，内核记录地址在 ntdll.dll 内部。

```
Direct (detectable):            Camouflaged (safe):
controller.exe                  controller.exe
  └─ syscall ← recorded here      └─ call ntdll!syscall;ret
                                                   └─ syscall ← recorded here
```

**5-arg syscall 特殊处理:** `call` 指令会 push 返回地址 (rsp-=8)，导致 arg5 从 [rsp+0x28] 偏移到 [rsp+0x30]。解决：`call` 前将 arg5 从 [rsp+0x28] 复制到 [rsp+0x20]。

**降级策略 / Fallback:** gadget 查找失败时 `gadget` 字段为 0，自动回退到直接 `syscall` 指令。

### NtOpenProcess 间接调用 / Indirect NtOpenProcess

不再使用 `windows` crate 的 `OpenProcess`。直接用间接 syscall 调用 `NtOpenProcess`，构造 `OBJECT_ATTRIBUTES` (全零, Length=48) + `CLIENT_ID` (填入 PID)。

**Bypasses kernel32!OpenProcess and ntdll!NtOpenProcess hooks entirely. Uses raw indirect syscall with constructed OBJECT_ATTRIBUTES and CLIENT_ID.**

### 进程枚举 / Process Enumeration

间接 `NtQuerySystemInformation(SystemProcessInformation=5)` → 解析 `SYSTEM_PROCESS_INFORMATION` 链表。替代了 `CreateToolhelp32Snapshot`。

**Replaces CreateToolhelp32Snapshot with indirect NtQuerySystemInformation. Parses SYSTEM_PROCESS_INFORMATION linked list for PID + ImageName extraction.**

### 模块枚举 / Module Enumeration (PEB Walk)

1. `NtQueryInformationProcess(ProcessBasicInformation=0)` → PEB 基址
2. 读取 `PEB.Ldr` → `PEB_LDR_DATA.InLoadOrderModuleList`
3. 遍历双向链表 → 读取 `LDR_DATA_TABLE_ENTRY` (DllBase, SizeOfImage, BaseDllName)
4. 全部通过间接 syscall + gadget 伪装

## 功能模块详情 / Feature Module Details

### Silent Aim (silent_aim.rs)

**原理:** 直接向游戏进程写入 `m_angEyeAngles`，不移动本地鼠标。**Writes view angles directly to server memory — no local mouse movement.**

流程：获取本地 Pawn 和当前视角 → 遍历 `C_CSPlayerPawn` 实体计算最佳瞄准角度 → 写入新角度 → 可选自动开枪 → 恢复原角度。

**注意:** 本模块是唯一一个**写入**游戏内存的 Enhancement。**The only module that writes to game memory.**

### Trigger Bot (trigger.rs)

通过 `CalcShotWidth` 获取准星精度 → 检测准星内实体 → 计算命中概率。支持随机延迟区间。

### Aim Assist Recoil (aim.rs)

读取 `m_aimPunchAngle` → 计算反向补偿 → 写入 `m_angEyeAngles`。最少子弹数阈值。

### ESP (player/ 目录)

骨骼渲染通过 `CSkeletonInstance` 骨骼数组。Box2D/Box3D 通过 `m_vecAbsOrigin` + `m_vecViewOffset` 计算。配置树递归渲染。

### Grenade Helper (grenade_helper.rs)

基于视野预测投掷物落点，圆环 + 角度可视化。

## 关键设计决策 / Key Design Decisions

### 为什么需要双后端？/ Why Dual Backend?

- **内核驱动**提供最强隐匿性：内核句柄表、ObRegisterCallbacks 保护、CR3 缓解
- **用户态后端**提供零门槛体验：不需要编译驱动、不需要重启、对 VAC 足够安全
- Syscall 返回地址伪装 + 间接 NtOpenProcess/NtQuerySystemInformation 使 VAC 难以检测

### 为什么 Silent Aim 用 `runtime_offset!` 而不是 trait 写方法？

Schema trait 只提供读取方法。写入需要直接计算偏移地址，用 `runtime_offset!` 保证与读取路径相同偏移量来源。

### Web 面板 vs ImGui Overlay？

- **ImGui Overlay**：游戏内实时显示，适合战斗中快速查看
- **Web 面板**：完整配置管理，适合战前/战后精细调整
- 二者共享同一份配置 (AppSettings)，通过 `settings_changed` AtomicBool 同步

## 代码规范 / Code Conventions

- 使用 `anyhow` 处理错误，`context()` 添加上下文
- 游戏结构体字段通过 schema trait 访问 (`pawn.m_iHealth()?`)
- 偏移量优先 schema trait，写入时用 `runtime_offset!`
- 内存指针 `Ptr64<T>` + `value_reference()` 安全解引用
- Web API 响应统一 `Json<Value>` 格式
- 后端通过 `DriverInterface` trait 多态，不直接依赖具体实现
