# LouisMod — 交接文档

## 项目概况

LouisMod 是一个外部、只读、内核级的 CS2 gameplay enhancer（基于 Valthrun 二次开发）。本 Fork 在原版基础上扩展了多项功能，并重新设计了 Web 控制面板。

**技术栈：**
- 语言：Rust (nightly toolchain)
- 驱动接口：`louismod-kdriver` — 自研内核驱动 + 用户态间接 syscall 双后端
- 双后端自动降级：内核驱动可用时使用驱动，不可用时自动回退用户态备用后端
- GUI Overlay：imgui-rs + 自定义 OpenGL 渲染
- Web 面板：Axum (HTTP/WS) + 纯前端 HTML/CSS/JS
- Schema 解析：自定义 `raw_struct` + 运行时偏移量解析

## 核心架构

### 0. 双后端系统（关键设计）

LouisMod 使用**双后端自动降级**架构。`create_driver()` 工厂函数按顺序尝试：

```
create_driver()
  ├── 1. LouisModDriver::create_from_env()  ── 内核驱动（需要 louismod.sys 已加载）
  └── 2. UserModeDriver::create()           ── 用户态间接 syscall（无驱动时自动回退）
```

**内核驱动后端 (LouisModDriver):**
- 通过 `CreateFileW("\\.\LouisMod")` + `DeviceIoControl` 与内核驱动通信
- 提供：进程列表、模块列表、内存读写、键盘/鼠标输入、进程保护(ObRegisterCallbacks)、CR3 缓解
- 需要：testsigning 开启 + 驱动已加载 + 管理员权限

**用户态后端 (UserModeDriver):**
- 使用间接 syscall（从磁盘 ntdll.dll 提取 syscall 号）
- syscall 返回地址伪装：`call` 到 ntdll 的 `syscall;ret` gadget，内核回溯看起来调用来自 ntdll.dll
- 进程枚举：间接 NtQuerySystemInformation(SystemProcessInformation)
- 模块枚举：PEB walk（通过间接 NtQueryInformationProcess + NtReadVirtualMemory）
- 句柄：间接 NtOpenProcess（绕过 kernel32/ntdll 函数入口 hook）
- 输入：SendInput（用户态唯一可行方案）
- 限制：无内核句柄隐藏、无进程保护、无 CR3 缓解
- 仅需：非管理员即可运行（前提是能 OpenProcess 目标 PID）

### DriverInterface trait

两套后端通过统一的 `DriverInterface` trait 向上层暴露相同接口：

```rust
pub trait DriverInterface: Send + Sync {
    fn driver_features(&self) -> DriverFeature;
    fn driver_version(&self) -> VersionInfo;
    fn total_read_calls(&self) -> usize;

    fn list_processes(&self) -> IResult<Vec<ProcessInfo>>;
    fn list_modules(&self, pid: ProcessId, dt: DirectoryTableType) -> IResult<Vec<ProcessModuleInfo>>;

    fn read_bytes(&self, pid: ProcessId, dt: DirectoryTableType, addr: u64, buf: &mut [u8]) -> IResult<()>;
    fn write_bytes(&self, pid: ProcessId, dt: DirectoryTableType, addr: u64, buf: &[u8]) -> IResult<()>;

    fn send_keyboard_state(&self, states: &[KeyboardState]) -> IResult<()>;
    fn send_mouse_state(&self, states: &[MouseState]) -> IResult<()>;

    fn toggle_process_protection(&self, mode: ProcessProtectionMode) -> IResult<()>;
    fn add_metrics_record(&self, rt: &str, rp: &str) -> IResult<()>;
}
```

上层 (cs2/handle.rs) 通过 `Arc<dyn DriverInterface>` 使用，无需关心底层是驱动还是用户态。

### DriverFeature 标志位

```rust
pub struct DriverFeature(u32);
// PROCESS_LIST | PROCESS_MODULES | MEMORY_READ | MEMORY_WRITE
// | INPUT_KEYBOARD | INPUT_MOUSE | METRICS
// 内核特有：PROCESS_PROTECTION_KERNEL | CR3_SHENANIGANS
```

用户态后端只报告其实际支持的特性。上层代码通过 `driver_features()` 查询能力。

### 1. Enhancement 系统

所有游戏功能都实现 `Enhancement` trait：

```rust
pub trait Enhancement {
    fn update(&mut self, ctx: &UpdateContext) -> anyhow::Result<()>;
    fn render(&self, states: &StateRegistry, ui: &imgui::Ui, unicode_text: &UnicodeTextRenderer) -> anyhow::Result<()>;
    fn render_debug_window(&mut self, ...) -> anyhow::Result<()> { Ok(()) }
}
```

- **update()** — 每帧调用，从游戏内存读取数据并执行逻辑（如写入角度、发送鼠标事件）
- **render()** — ImGui 渲染，用于 Overlay 显示
- **render_debug_window()** — 调试窗口（可选）

所有 Enhancement 在 `main.rs` 中注册并循环调用。

### 2. 状态注册表 (StateRegistry)

```rust
let mut app_state = StateRegistry::new(1024 * 8);
app_state.register::<StateCS2Memory>()?;
app_state.register::<StateEntityList>()?;
// ...
```

各模块通过 `ctx.states.resolve::<T>(())?` 获取共享状态。StateRegistry 内部使用 `any` 类型擦除存储。

### 3. CS2Handle 与内存读写

`CS2Handle` 封装了与后端的通信：
- `read_sized<T>(address)` — 读取任意类型（小类型用栈缓冲区 / 大类型用堆）
- `read_slice<T>(address, buffer)` — 批量读取（≤256 字节用栈缓冲区，超过用堆）
- `write_sized<T>(address, value)` — 写入游戏内存（仅 silent_aim 使用）
- `send_mouse_state()` / `send_keyboard_state()` — 发送输入事件
- `create_memory_view()` — 创建 MemoryView 供 schema trait 读取

### 4. Schema 与偏移量解析

**两种偏移量获取方式：**

1. **Trait 方法**（推荐读取）：
   ```rust
   let angles = pawn.m_angEyeAngles()?;  // 通过 schema trait 动态查偏移
   ```

2. **`runtime_offset!` 宏**（写入时使用）：
   ```rust
   let offset = runtime_offset!(15824, "client.dll", "C_CSPlayerPawn", "m_angEyeAngles");
   ```
   第一个参数是 fallback 默认值，后续是 `(模块, 类, 成员)` 三元组。
   宏内部创建 `static mut Option<u64>` 缓存，首次调用通过全局 `PROVIDER_INSTANCE` 的 `BTreeMap` 查找，
   后续直接返回缓存值，零开销。

## 用户态后端详解

### Syscall 号提取

`syscall::load_syscall_table()` 从磁盘读取 `C:\Windows\System32\ntdll.dll`，解析 PE 结构 + export 目录，提取 5 个关键函数的 syscall 号：

| 函数 | 用途 |
|------|------|
| NtReadVirtualMemory | 跨进程读取 |
| NtWriteVirtualMemory | 跨进程写入 |
| NtQuerySystemInformation | 进程枚举 |
| NtOpenProcess | 打开进程句柄 |
| NtQueryInformationProcess | 获取 PEB 地址 |

提取方法：定位 export 中的函数 RVA → 检查 stub 的第一个字节是否为 `0xB8`（mov eax, imm32）→ 读取 imm32 即为 syscall 号。

### Syscall 返回地址伪装（Gadget Camouflage）

**问题：** 直接在我们的模块执行 `syscall` 指令，内核回溯显示调用来自 `controller.exe`，反作弊系统可以检测。

**解决方案：** `find_syscall_ret_gadget()` 在运行时已加载的 ntdll.dll 的 `.text` 段中搜索 `0F 05 C3`（`syscall; ret`）字节序列。找到后，通过 `call` 指令跳转到该地址执行 syscall，内核记录的 syscall 指令地址在 ntdll.dll 内部，调用链看起来完全正常。

```
我们的代码                         ntdll.dll
───────────                      ──────────
mov r10, rcx
mov eax, sysnum                  syscall    ← 内核记录的调用地址在这里
call gadget ─────────────────→   ret        → 返回我们
; 继续执行
```

**5 参数 syscall 的特殊处理：** `call` 指令会 push 返回地址（rsp-=8），导致 arg5 从 [rsp+0x28] 偏移到 [rsp+0x30]。解决方案：在 `call` 之前将 arg5 从 [rsp+0x28] 复制到 [rsp+0x20]，call 后 gadget 看到的 arg5 恰好在 [rsp+0x28]。

**降级策略：** 如果 gadget 查找失败（某些极端环境），`gadget` 字段为 0，`syscall_*_via_gadget()` 自动回退到直接 `syscall` 指令。

### NtOpenProcess 间接调用

不再使用 `windows` crate 的 `OpenProcess`（走 kernel32→ntdll 标准路径，可能被 hook）。直接用间接 syscall 调用 `NtOpenProcess`：

```rust
pub fn nt_open_process_via_gadget(gadget, syscall_num, pid, desired_access) -> IResult<u64>
```

参数构造：`OBJECT_ATTRIBUTES`（全零，仅 Length=48）、`CLIENT_ID`（填入目标 PID）。

### 进程枚举

通过间接 `NtQuerySystemInformation(SystemProcessInformation=5)` 获取所有进程，解析 `SYSTEM_PROCESS_INFORMATION` 链表。替代了 `CreateToolhelp32Snapshot` / `Process32FirstW`，避免走 Win32 API。

### 模块枚举 (PEB Walk)

1. `NtQueryInformationProcess(ProcessBasicInformation=0)` → 获取 PEB 基址
2. 读取 `PEB.Ldr` → `PEB_LDR_DATA.InLoadOrderModuleList`
3. 遍历双向链表 → 读取每个 `LDR_DATA_TABLE_ENTRY`（DllBase, SizeOfImage, BaseDllName）
4. 全部通过间接 syscall（可伪装 gadget）进行

## Web 控制面板

### 服务器

基于 **Axum 0.7**，运行在端口 `16384`，支持 HTTP + WebSocket。

**路由：**
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/` | 返回 index.html（编译时嵌入） |
| GET/PUT | `/api/settings` | 读取/更新全部设置 |
| GET | `/api/settings/save` | 保存设置到 config.yaml |
| GET | `/api/esp-configs` | 获取所有 ESP 配置 + 配置树 |
| GET/PUT | `/api/esp-config/{key}` | 读取/更新单个 ESP 配置 |
| GET | `/api/action/toggle-feature` | 一键切换功能（兼容旧版） |
| WS | `/ws` | WebSocket 实时推送设置变更 |

### WebSocket 协议

连接后服务器立即发送初始设置：
```json
{"type":"settings","payload":{...}}
```

设置变更时广播相同格式消息。客户端可发送 `{"type":"ping"}` 保活。

### 设置序列化

**关键细节：** `GrenadeSettings` 使用 `#[serde(flatten, with = "serde_prefix_grenade_helper")]`，
所有字段以 `grenade_helper_` 前缀展平到根对象。前端访问示例：
```javascript
// ❌ 错误（旧前端写法）
settings.grenade_helper.active

// ✅ 正确（展平后）
settings.grenade_helper_active
```

**ESP 配置序列化：**
```json
{
  "type": "Player",
  "box_type": "Box2D",
  "skeleton": true,
  "health_bar": true,
  ...
}
```

**Color 序列化：**
- `EspColor`：`{"type": "Static", "options": {"value": 4294967295}}`（u32 RGBA LE）
- GrenadeHelper Color：`[r, g, b, a]`（f32 数组，0.0–255.0）

### 前端架构

`index.html` 是单页应用，包含：
- **页面系统**：仪表板 / ESP / Trigger Bot / Silent Aim / Grenade Helper / 设置
- **实时更新**：WebSocket 监听设置变更
- **命令面板**：Ctrl+K 打开
- **配置文件**：localStorage 保存/载入，JSON 导入/导出

- `PUT /api/settings`：先尝试完整反序列化（未指定的字段→默认值），失败后 fallback 到 deep merge

## 功能模块详情

### Silent Aim (silent_aim.rs)

**原理：** 直接向游戏进程写入 `m_angEyeAngles`，不移动本地鼠标。

**关键流程：**
1. 获取本地玩家 Pawn 和当前视角角度
2. 遍历所有 `C_CSPlayerPawn` 实体，计算最佳瞄准角度
3. 写入新角度 → 可选自动开枪 → 立即恢复原角度

**写入偏移量：** 使用 `runtime_offset!` 动态解析，确保与读取路径一致。

**注意：** 本模块是唯一一个**写入**游戏内存的 Enhancement（非只读）。

### Trigger Bot (trigger.rs)

- 通过 `CalcShotWidth` 函数获取当前准星精度
- 检测准星内实体，计算命中概率
- 支持随机延迟区间（min/max delay）

### Aim Assist Recoil (aim.rs)

- 读取 `m_aimPunchAngle` 获取后坐力
- 计算反向补偿角度并写入 `m_angEyeAngles`
- 最少子弹数后才启用

### ESP (player/ 目录)

- 骨骼渲染通过 `CSkeletonInstance` 的骨骼数组
- Box2D/Box3D 通过 `m_vecAbsOrigin` + `m_vecViewOffset` 计算
- ESP 配置树（EspSelector）：Player → PlayerTeam → PlayerTeamVisibility，递归渲染

### Grenade Helper (grenade_helper.rs)

- 基于视野预测投掷物落点
- 圆环 + 角度可视化
- 预设投掷物点位管理

## 关键设计决策

### 为什么 Silent Aim 使用 `runtime_offset!` 而不是 trait 写方法？

Schema 生成的 trait 只提供了读取方法。写入操作需要直接计算偏移地址，
使用 `runtime_offset!` 保证与读取路径使用相同的偏移量来源（全局 SchemaProvider 的 BTreeMap），
避免硬编码失效。

### 为什么是只读架构？

原版 Valthrun 的核心设计哲学是只读——不写入游戏进程，因而不可能被内存扫描检测。
Silent Aim 打破了这一原则（需要写入角度），但限制在最小必要范围。

### 为什么需要双后端？

- **内核驱动后端**提供最强的隐匿性：句柄在内核句柄表中、ObRegisterCallbacks 保护、CR3 缓解
- **用户态后端**提供"零门槛"体验：不需要编译驱动、不需要重启、不需要管理员权限，对 VAC 足够安全
- syscall 返回地址伪装和间接 NtOpenProcess/NtQuerySystemInformation 使 VAC 难以检测

### Web 面板 vs ImGui Overlay？

- **ImGui Overlay**：游戏内实时显示，适合战斗中快速查看/调整
- **Web 面板**：完整的配置管理，适合战前/战后精细调整

二者共享同一份配置（AppSettings），通过 `settings_changed` AtomicBool 同步。

## 代码规范

- 使用 `anyhow` 处理错误，`context()` 添加上下文
- 游戏结构体字段通过 schema trait 访问（`pawn.m_iHealth()?`）
- 偏移量优先使用 schema trait，写入时用 `runtime_offset!`
- 内存指针使用 `Ptr64<T>` + `value_reference()` 安全解引用
- Web API 响应统一使用 `Json<Value>` 格式
- 后端代码通过 `DriverInterface` trait 多态，不直接依赖具体实现
